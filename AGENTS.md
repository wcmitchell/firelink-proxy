# AGENTS.md

## Project Overview

Firelink Proxy is the ingress layer for Firelink, a web GUI for Bonfire. It is a single OpenShift
pod running two containers in a sidecar pattern: an OpenShift OAuth Proxy that handles
authentication and TLS termination, and a Caddy reverse proxy that routes `/api/*` requests to the
[Firelink backend][firelink-backend] and all other requests to the
[Firelink frontend][firelink-frontend]. The proxy has no application code — it is a
configuration-only infrastructure component.

## Dependencies

**Runtime:** Caddy (UBI-based image), OpenShift OAuth Proxy (ose-oauth-proxy-rhel9)

**Build:** Docker/Buildah

## Development Commands

See [Deploying][readme-deploying] in the README for the full deployment reference.

```bash
docker build -t firelink-proxy:latest .   # Build the container image
./build_deploy.sh                          # Build, tag with git hash, push to Quay
```

CI runs Tekton pipelines (`.tekton/`) that build the container image and run security scans (Clair,
ClamAV, Snyk SAST, ShellCheck). There are no tests — this is a configuration-only project.

## Architecture

The pod runs an OAuth Proxy sidecar on port 8888 (HTTPS) that authenticates users and forwards to
a Caddy container on port 8000 (HTTP). Caddy routes requests to backend and frontend services using
environment variables for service discovery. A dedicated route for `/api/firelink/socket.io/*`
injects WebSocket upgrade headers.

See [ARCHITECTURE.md][architecture] for the full sidecar container design, OAuth Proxy
configuration, Caddy routing rules, and deployment model.

## Code Style

- No linter or formatter is configured
- The Caddyfile uses Caddy's native configuration syntax with `{env.VAR}` interpolation for
  environment-based service discovery
- Shell scripts follow Bash conventions; ShellCheck runs in CI via the Tekton pipeline

## Common Mistakes

- **WebSocket route ordering.** The `/api/firelink/socket.io/*` route must appear before the
  general `/api/*` route in the Caddyfile. Caddy evaluates routes in declaration order. If the
  Socket.IO route comes second, the `/api/*` rule matches first and WebSocket upgrade headers are
  not injected — Socket.IO falls back to HTTP long-polling silently.
- **Environment variable dependency.** The Caddyfile references `BACKEND_SERVICE`, `BACKEND_PORT`,
  `FRONTEND_SERVICE`, and `FRONTEND_PORT` via `{env.VAR}` syntax. If these are not set, Caddy will
  start but route requests to empty hostnames, producing opaque 502 errors. Locally, these must be
  exported before running the container.
- **TLS certificate provisioning.** The deployment template expects a `firelink-proxy-tls` Secret
  auto-generated by the OpenShift serving cert signer (via the
  `service.alpha.openshift.io/serving-cert-secret-name` annotation on the Service). If the cluster
  does not have the serving cert signer enabled, the OAuth Proxy container will fail to start with a
  TLS certificate error. The separate `firelink-proxy` Secret (OAuth cookie secret) must also exist.
- **OAuth redirect reference.** The ServiceAccount has an `oauth-redirectreference.primary`
  annotation pointing to the Route named `firelink-proxy`. If the Route is renamed or not yet
  created when the OAuth flow triggers, authentication will fail with a redirect mismatch error.

## Deployment

The app is containerized with a Dockerfile that copies the Caddyfile into a UBI-based Caddy image.
The deployment template (`deploy/deploy.yaml`) creates a Deployment with two sidecar containers,
a Service, ServiceAccount, Route, and ClusterRoleBinding. Tekton pipelines in `.tekton/` build and
push images to Quay on every push to `master`.
