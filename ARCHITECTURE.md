# Architecture

## System Overview

Firelink Proxy is the ingress layer for [Firelink][firelink-frontend], a web GUI for
[Bonfire][bonfire]. It is a single OpenShift pod running two containers in a sidecar pattern: an
OpenShift OAuth Proxy that handles authentication and TLS termination, and a Caddy reverse proxy
that routes authenticated requests to the [frontend][firelink-frontend] and
[backend][firelink-backend] services. This ensures both the frontend SPA and backend API are behind
OAuth without either needing to implement the OAuth flow themselves.

### Firelink Ecosystem

```
Browser  -->  OpenShift Route (:8888 TLS)
                |
          OAuth Proxy sidecar (:8888)          ── firelink-proxy pod ──
                |                              |                      |
          Caddy reverse proxy (:8000)          |  two containers in   |
              /   \                            |  a single pod        |
             /     \                           ── (sidecar pattern) ──
            v       v
  /api/*  -->  firelink-backend service  -->  OpenShift / Bonfire
                                         -->  Prometheus
  /*      -->  firelink-frontend service -->  app-interface (GraphQL)
               (React SPA, own Caddy)
```

### Request Flow

1. A browser request arrives at the OpenShift Route on port 8888 over TLS (reencrypt termination).
2. The OAuth Proxy validates the user's OpenShift session. Unauthenticated users are redirected to
   the OpenShift OAuth login flow. Requests to `/metrics` bypass authentication
   (`-skip-auth-regex=^/metrics`).
3. The OAuth Proxy injects user identity headers (`gap-auth` containing the user's email, plus the
   bearer token via `-pass-user-bearer-token=true`) and forwards the request to Caddy on
   `localhost:8000`.
4. Caddy routes the request based on path:
   - `/api/firelink/socket.io/*` — forwarded to the backend service with WebSocket upgrade headers
     (`Connection`, `Upgrade`) injected.
   - `/api/*` — forwarded to the backend service.
   - All other paths — forwarded to the frontend service.

## Technology Stack

- **Reverse proxy**: Caddy (UBI-based image)
- **Authentication**: OpenShift OAuth Proxy (ose-oauth-proxy-rhel9)
- **Container platform**: OpenShift
- **CI/CD**: Tekton pipelines via Konflux

## Container Architecture

The pod runs two containers that communicate over localhost:

### OAuth Proxy Container

The OpenShift OAuth Proxy listens on port 8888 (HTTPS) and handles:

- OpenShift OAuth authentication flow (`-provider=openshift`)
- TLS with a serving certificate from the `firelink-proxy-tls` Secret (auto-generated by the
  OpenShift serving cert signer via the
  `service.alpha.openshift.io/serving-cert-secret-name` annotation)
- Cookie-based session management with a client secret from the `firelink-proxy` Secret
- User identity header injection (`-pass-user-headers=true`), which sets the `gap-auth` header
  containing the authenticated user's email address
- Bearer token passthrough (`-pass-user-bearer-token=true`)
- Authorization delegation via `system:auth-delegator` ClusterRoleBinding, requiring that
  authenticated users can `get pods` in the namespace
  (`-openshift-delegate-urls={"\/":{"group":"","resource":"pods","verb":"get"}}`)

The upstream target is `http://localhost:8000` — the Caddy container in the same pod.

### Caddy Container

Caddy listens on port 8000 (HTTP, auto-HTTPS disabled) and routes requests to backend and frontend
services using environment variables for service discovery:

| Route pattern | Target | Headers added |
|---|---|---|
| `/api/firelink/socket.io/*` | `BACKEND_SERVICE:BACKEND_PORT` | `Connection`, `Upgrade`, `Host`, `X-Real-IP`, `X-Forwarded-For`, `X-Forwarded-Proto` |
| `/api/*` | `BACKEND_SERVICE:BACKEND_PORT` | (default proxy headers) |
| `/*` (catch-all) | `FRONTEND_SERVICE:FRONTEND_PORT` | (default proxy headers) |

The WebSocket route is listed first because Caddy evaluates routes in declaration order and the
`/api/*` rule would otherwise consume Socket.IO traffic without the upgrade headers needed for
WebSocket connections. The frontend and backend are separate services with their own pods — Caddy
does not serve any static files.

## Environment Variables

| Variable | Default | Description |
|---|---|---|
| `BACKEND_SERVICE` | `firelink-backend` | Kubernetes service name for the backend |
| `BACKEND_PORT` | `8000` | Port the backend service listens on |
| `FRONTEND_SERVICE` | `firelink-frontend` | Kubernetes service name for the frontend |
| `FRONTEND_PORT` | `8000` | Port the frontend service listens on |

These are set via the OpenShift deployment template parameters and injected as container environment
variables. Caddy reads them at startup using `{env.VAR}` syntax in the Caddyfile.

## Deployment

### OpenShift Template

The deployment template (`deploy/deploy.yaml`) creates five resources:

1. **Deployment** — A single-replica pod with the two sidecar containers described above. The OAuth
   Proxy mounts TLS and client secret volumes; the Caddy container receives backend/frontend
   service coordinates as environment variables.
2. **Service** — Exposes port 8888 (the OAuth Proxy) as the pod's entry point.
3. **ServiceAccount** — `firelink-proxy`, annotated with an `OAuthRedirectReference` pointing to
   the Route so the OAuth login flow redirects back correctly.
4. **Route** — TLS-reencrypt route with cert-manager annotations for automatic certificate
   provisioning via the `letsencrypt-devshiftnet-dns` ClusterIssuer. The hostname is
   `firelink.${INGRESS_DOMAIN}`.
5. **ClusterRoleBinding** — Grants the ServiceAccount `system:auth-delegator` so the OAuth Proxy
   can perform authorization checks.

A separate `deploy/secret.yaml` provides a sample `firelink-proxy` Secret with the OAuth cookie
client secret. In non-ephemeral environments, this Secret is provisioned externally.

### CI/CD with Tekton

Tekton pipelines in `.tekton/` automate the build on push to `master` and on pull requests:

- **Pull request pipeline** — Builds the container image, runs security scans (SAST, Clair, ClamAV,
  ShellCheck), and generates an SBOM. Built images expire after 5 days.
- **Push pipeline** — Same steps as the PR pipeline but produces permanent images pushed to
  `quay.io/redhat-user-workloads/hcm-eng-prod-tenant/firelink/firelink-proxy` and applies release
  tags.

### Local Build

The `build_deploy.sh` script builds the image locally with Docker, tags it with the short git
commit hash, and pushes to `quay.io/cloudservices/firelink-proxy`. It requires `QUAY_USER`,
`QUAY_TOKEN`, `RH_REGISTRY_USER`, and `RH_REGISTRY_TOKEN` environment variables for registry
authentication.

## Key Design Decisions

- **Sidecar pattern over middleware.** Authentication is handled by a separate container (OAuth
  Proxy) rather than middleware inside the backend or frontend. This keeps both application
  components authentication-free and allows the proxy to be updated, scaled, or replaced
  independently.
- **Caddy over Nginx/HAProxy.** Caddy was chosen for its minimal configuration syntax and native
  environment variable interpolation (`{env.VAR}`), which allows the same Caddyfile to work across
  environments without templating.
- **Explicit WebSocket route.** The `/api/firelink/socket.io/*` route is defined separately from
  the general `/api/*` route to inject `Connection` and `Upgrade` headers. Without these, Caddy's
  default reverse proxy behavior strips upgrade headers and WebSocket connections fail silently —
  the Socket.IO client falls back to HTTP long-polling, which works but with higher latency.
- **No health endpoint.** The proxy does not expose its own health check. Liveness and readiness
  probes rely on the OAuth Proxy's TLS listener being reachable on port 8888. The backend's
  `/health` endpoint verifies OpenShift API connectivity independently.
- **Environment-based service discovery.** Backend and frontend service names are injected as
  environment variables rather than hardcoded. This allows the same image to work in any namespace
  where the services may have different names (e.g., Clowder-managed environments append suffixes).
